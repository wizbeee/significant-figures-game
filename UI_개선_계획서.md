### 🟡 미흡하거나 개선이 필요한 영역

| # | 영역 | 현재 문제 | 영향 |
|:-:|------|----------|------|
| 1 | **페이지네이션 없음** | 조회 결과 전체를 한 번에 테이블에 렌더링. 100건 이상이면 DOM 성능 저하 | 교사가 느린 로딩을 경험 |
| 2 | **통계 카드 2중 호출** | `loadApplications()`에서 데이터 로드 후 `applyStats()`에서 동일 조건으로 서버 재호출 | 불필요한 서버 부하 |
| 3 | **메인 대시보드 헤더 용어** | "과학실험실 관리 시스템" → 향후 실습실 포함 시 맞지 않음 | 용어 일관성 |
| 4 | **필터 컨트롤 영역 복잡** | 5줄로 된 컨트롤 영역이 화면 절반을 차지. 정렬 4순위가 항상 보임 | 교사가 압도감 느낌 |
| 5 | **조회 결과 건수 미표시** | 테이블 위에 "총 N건 조회됨" 같은 정보 없음 | 결과 규모 파악 어려움 |
| 6 | **상세 모달에서 편집 전환 어색** | 상세 보기 → "전체 편집" 클릭 → 별도 편집 모달. 상세에서 바로 수정 불가 | 2번 클릭 필요 |
| 7 | **시약 상세에 폐기 방법 미표시** | 상세 모달의 시약 테이블에 '폐기 방법' 열 없음 | 관리자가 폐기 정보 확인 불가 |
| 8 | **테이블 열 고정 너비** | `table-layout: fixed` + `white-space: nowrap` → 긴 내용 잘림 | 실험 제목 등 내용 확인 어려움 |
| 9 | **실험실 옵션 불일치** | admin_body의 필터 옵션과 실제 데이터의 실험실명이 불일치 가능 (예: `Tecｈ&Art LaB` 전각 문자) | 필터가 작동하지 않는 실험실 존재 |
| 10 | **부가 페이지 네비게이션** | 시약/학생/임장 페이지에서 "← 관리자 페이지" 링크만 있음. 페이지 간 이동 불가 | 매번 메인으로 돌아가야 함 |
| 11 | **삭제 시 확인 절차 미흡** | `deleteOne()`이 confirm만으로 삭제 진행. 되돌리기(Undo) 없음 | 실수로 삭제 시 복구 불가 |
| 12 | **조건 초기화 후 데이터 사라짐** | "전체 조건 초기화" 클릭 시 VIEW가 비워지고 "조회 버튼을 클릭하세요" 표시 | 초기화 후 다시 조회 버튼 클릭 필요 |
| 13 | **saveAllEdits 직렬 처리** | 변경 건을 1건씩 순차 저장. 50건이면 50번 서버 호출 | 저장 시간 매우 길어짐 |
| 14 | **학생 명단 출석 체크 미저장** | 출석 체크란이 있지만 체크 결과를 저장하는 서버 함수 없음 | 인쇄용으로만 사용 가능 |
| 15 | **모바일 대응 부족** | 메인 대시보드에 `@media` 반응형 CSS 없음 | 모바일에서 컨트롤 영역 깨짐 |

# UI 개선 계획

## Phase 1: 핵심 UX 개선 (높은 우선순위)

### P1-1. 컨트롤 영역 접이식(Collapsible) 구조화

**현재**: 5줄 컨트롤이 항상 펼쳐져 화면 절반 차지
**개선**:
- **1줄 (항상 표시)**: 기간 + 빠른 기간 + 조회 + 전체 초기화
- **2줄 (항상 표시)**: 실험실 + 상태 + 검색
- **3줄 이하 (접이식)**: "고급 옵션 ▼" 토글 버튼 → 클릭 시 정렬/편집모드/기타 확장
- 접이식 상태를 `localStorage`에 저장하여 재방문 시 유지

```
┌─────────────────────────────────────────────────┐
│ 기간: [2026-04-01] ~ [2026-04-07]  오늘|이번주  [조회] [초기화] │
│ 실험실: [전체▼] 상태: [전체▼] 검색: [________]              │
│                                                               │
│ [▼ 고급 옵션]                                                │
│ ┌─────────────────────────────────────────────┐             │
│ │ 정렬 1순위: [제출일시▼][오름▼]  2순위: ...    │             │
│ │ [편집모드] [일괄 변경 바...]                  │             │
│ └─────────────────────────────────────────────┘             │
└─────────────────────────────────────────────────┘
```

**기대 효과**:
- 일반 조회 시 화면 공간 40% 절약 → 데이터 테이블 영역 확대
- 교사가 자주 쓰는 기능(기간/실험실/상태)만 노출 → 학습 곡선 감소
- 고급 기능(정렬/편집)은 필요할 때만 펼침 → 초보 사용자 압도감 해소

### P1-2. 테이블 페이지네이션 + 건수 표시

**현재**: 조회 결과 전체를 한 번에 렌더링
**개선**:
- 테이블 상단에 "총 N건 조회됨 (N페이지)" 표시
- 기본 50건씩 표시, 하단에 페이지네이션 버튼
- 건수 선택: 25 | 50 | 100 | 전체
- 페이지 이동 시 체크 상태(SELECTED) 유지

```
총 127건 조회됨  |  25건 | [50건] | 100건 | 전체  |  ◀ 1 2 [3] 4 5 ▶
┌─────────────────────────────────────────────────┐
│ ☐ │ 제출일시 │ 대표자 │ 실험실 │ ...            │
│ ☐ │ 2026-04-05 │ 홍길동 │ 화학실험실 │ ...     │
│ ...                                              │
└─────────────────────────────────────────────────┘
◀ 1 2 [3] 4 5 ▶
```

**기대 효과**:
- 100건 이상 데이터에서 렌더링 속도 80% 개선 (DOM 노드 감소)
- 조회 결과 규모를 즉시 파악 가능
- 페이지 간 이동이 빠르고 직관적

### P1-3. 통계 카드 클라이언트 계산 + 실시간 연동

**현재**: `applyStats()`가 서버에 별도 요청하여 통계 로드
**개선**:
- `loadApplications()`에서 받은 ALL 데이터로 클라이언트에서 직접 통계 계산
- 필터 변경 시 VIEW 기준으로 통계 실시간 갱신
- 서버 `getApplicationStats()` 호출 제거

```js
function updateStatsFromView() {
  const stats = { total: ALL.length, pending: 0, firstApproved: 0, finalApproved: 0, rejected: 0 };
  ALL.forEach(app => {
    const s = calcStatus(app);
    if (s === '대기') stats.pending++;
    else if (s === '1차승인') stats.firstApproved++;
    // ...
  });
  // 필터링된 VIEW 기준 추가 표시도 가능
}
```

**기대 효과**:
- 서버 요청 50% 감소 (데이터 로드 1회로 통합)
- 필터 변경 시 통계가 즉시 반영 → 실시간 대시보드 느낌
- 네트워크 지연으로 인한 통계 깜빡임 제거

### P2-4. 삭제 시 소프트 삭제 + 되돌리기

**현재**: 삭제 시 confirm → 즉시 행 삭제 (되돌리기 불가)
**개선**:
- 삭제 시 실제 삭제 대신 '삭제됨' 상태로 변경 (소프트 삭제)
- 테이블 하단에 "방금 1건 삭제됨 [되돌리기]" 스낵바 5초간 표시

- 5초 경과 또는 페이지 이탈 시 실제 삭제 실행

또는 (더 간단한 방안):
- confirm 대화상자를 2단계로 강화: "정말 삭제하시겠습니까?" + 신청ID 재입력 확인
- 삭제 후 스낵바로 결과 알림

**기대 효과**:
- 실수로 인한 데이터 손실 방지
- 삭제 작업에 대한 안전망 제공

### P2-5. saveAllEdits 배치 저장 최적화

**현재**: 변경 건을 1건씩 순차 `updateApplicationFields()` 호출
**개선**:
- 서버에 `batchUpdateApplications(updates[])` 함수 추가
- 클라이언트에서 모든 변경 사항을 하나의 배열로 모아 1회 서버 호출
- 서버에서 LockService로 보호하면서 일괄 처리

```js
// Code.gs
function batchUpdateApplications(updatesList) {
  // updatesList: [{id, changes}, {id, changes}, ...]
  const lock = LockService.getScriptLock();
  lock.tryLock(30000);
  try {
    updatesList.forEach(u => {
      updateApplicationFieldsInternal_(u.id, u.changes);
    });
    return { ok: true, count: updatesList.length };
  } finally {
    lock.releaseLock();
  }
}
```

**기대 효과**:
- 50건 저장 시 서버 호출 50회 → 1회로 감소
- 저장 시간 90% 단축 (네트워크 왕복 제거)
- 동시성 보호도 일괄 처리로 더 안정적

- 레이아웃 깨짐 방지

### P3-2. 실험실 옵션 서버 동기화

**현재**: 실험실 목록이 HTML에 하드코딩 (admin_body, bulkbar 등)
**개선**:
- 서버에서 실험실 목록 제공 (`getLabList()`)

### Before (현재)
```
교사 A가 오늘의 실험 현황을 확인하려 함:
1. 로그인 → 대시보드 진입
2. 기간 설정 → 조회 클릭 → 전체 데이터 로딩 대기
3. 통계 카드 별도 로딩 대기
4. 컨트롤 영역이 화면 절반 차지 → 스크롤하여 데이터 확인
5. 상태 필터링 → 어떤 필터가 적용되었는지 헷갈림
6. 상세 보기 → "전체 편집" → 수정 → 저장
7. 학생 명단 확인 → "← 관리자" → "👥 학생 명단" 클릭
총 소요 시간: ~3분, 클릭 수: ~15회
```

```
교사 A가 오늘의 실험 현황을 확인하려 함:

2. "오늘" 빠른 선택 → 조회 → 데이터+통계 동시 로딩

6. 네비게이션 바에서 "👥 학생 명단" 직접 이동
총 소요 시간: ~1.5분, 클릭 수: ~8회
```
